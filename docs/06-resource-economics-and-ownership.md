## 1. 五资源语义与不可替代性

### 1.1 资源清单

| 资源 | 中文 | 性质 | Phase 启用 |
|---|---|---|---|
| `energy` | 能量 | 行动客观储备，由 token 翻译产生消耗 | Phase 1 起 |
| `wealth` | 财富 | 经济媒介，多源获取 | Phase 3 起 |
| `knowledge` | 知识 | 客观知识量，与 SemanticConfirmed 同步 | Phase 1 起（计量），Phase 4 起（可兑） |
| `reputation` | 声望 | 数字社会信用 | Phase 4 起 |
| `social` | 社交资源 | 关系密度 / 社交资本 | Phase 5 起 |

Phase 未启用的资源**字段存在但值为 0**，不参与任何流通。资源生命周期由 `09` 路线明确。

### 1.2 不可替代性（V0.2 锁定）

**五资源彼此不可直接兑换**。例外仅以下三类（详见对应章节）：

| 例外 | 形式 | 限制 |
|---|---|---|
| `wealth → 世界服务进入` | 在学校 / 图书馆消耗 wealth → 获得 knowledge 增长**机会**（不是直接换 knowledge） | 服务介导，非兑换 |
| `social → reputation`（Phase 5+） | 长期高 social 经活动可缓慢提升 reputation | 受活动方式约束 |
| `token → energy`（系统内置） | LLMAdapter 内置翻译 | 唯一翻译点，详 §2 |

不可替代性是哲学约束 —— 防止资源体系退化为单一货币（=变相 token 计费）。

### 1.3 不可替代性的张力

某些场景下用户会要求"我用钱让生命体获得知识" —— 这是合理需求。V0.2 的处理：**通过世界服务介导**（用户充 wealth → 生命体去图书馆 → 学习并获得 knowledge 增长机会）。世界服务消耗 wealth 是合法的；直接 wealth → knowledge 兑换不合法。

登记 `R24 · 五资源不可互换的边界是否过严`。

## 2. Token 永不直接暴露原则

### 2.1 覆盖范围

**Token 数永不向以下方暴露**：

- UI Ecosystem（任何 UI）
- 用户（任何模式）
- 第三方 SDK 接入方
- Marketplace 参与方
- 运维 / 平台审计（详见 §9）

唯一可见 token 的代码点：`LLMAdapter` 内部（`04 §3.2`）。

### 2.2 翻译规则（V0.2 锁定）

```
energy_consumed = token_count × ModelCostFactor(model) × TimeWeight
```

- `ModelCostFactor`：不同 LLM 模型成本系数（GPT-4 > GPT-mini > 本地小模型）。
- `TimeWeight`：避免短时间密集调用过度耗 energy 的平滑权重。
- 翻译公式具体形状（线性 / 阶梯 / 上限）由 Phase 1 原型阶段标定。登记 `R25`。

### 2.3 用户可见的资源信号

UI 可呈现：

- `energy` 当前余额（0–1 归一化或抽象等级）
- `energy` 近期消耗速率（"今天消耗较快"）
- 当前 LLM 配置下的**预估可运行时长**（"按当前节拍约 N 天 / N 小时"）
- LLM 切换的"性价比"提示（"切换到 X 模型可让 energy 持续更久"）

UI 不可呈现：

- 任何 token 计数
- LLM 调用次数 / prompt 全文
- 任何与 token 1:1 关联的"次数 / 配额"

### 2.4 反模式（明令禁止）

| 反模式 | 为何禁止 |
|---|---|
| "剩余对话 X 次" | 把 token 计数包装为"对话次数" |
| "本次回复消耗 Y token" | 直接 token 暴露 |
| "充值 Z 元 = 100 万 token" | wealth 退化为 token 兑换券 |
| "今天 LLM 用了 N 次" | 间接 token 计数 |

### 2.5 双经济强解耦（V0.2.1 宪法补强）

Mindverse 显式存在**两层经济**，必须严格解耦：

| 层 | 量纲 | 谁可见 |
|---|---|---|
| **外部经济** | token / 配额 / 套餐 / 用户余额 / 模型单价 / 充值流水 | 平台运营 / 用户账户面板 / LLMAdapter 内部 |
| **内部经济** | energy / EnergyDailyCap / wealth / knowledge / reputation / social | Agent 子模块 / SDK / UI 生命体面板 |

**桥接规则**（详 `04 §3.2.1`）：

- 外部 token 消耗 → 内部 `energy` 增量消耗（LLMAdapter 翻译）
- 用户在账户面板设 token 限额 → LLMAdapter 翻译为 `LifeState.EnergyDailyCap`
- 内部 energy 余额**不反向映射**回 token
- Agent **不感知**外部经济量纲（详 `10 E06` 反模式）

**用户配置 EnergyDailyCap 的接口**：

- 用户在账户面板设置 `日 token 限额 / 5 小时窗口限额 / 月 wealth 预算` 等外部参数
- LLMAdapter 翻译并写入 `LifeState.EnergyDailyCap`（通过 StateManager）
- 调整频率有冷却期（详 `10 R46`）防止生命体"对自身规律的信任感"被频繁打扰

**外部经济耗尽分层处理**（Agent 无感知）：

| 场景 | 处理 |
|---|---|
| 日精力上限耗尽（EnergyDailyCap 达底） | Agent 自然看到 energy 低 → 节拍下调 → 拟生物疲惫 |
| 账户月套餐耗尽 | LLMAdapter 静默切到经济模型 / 本地 OSS；账户面板独立通知 |
| 所有 LLM 不可用 | LLMOffline → Dormant（`03 §5`）|

### 2.6 gas 不暴露原则（V0.2.1）

链上 `gas`（MindChain 操作费）与 LLM `token` 同等性质：**永不直接暴露给 Agent 与 UI**。

- **唯一可见点**：`BlockchainAdapter` 内部（`04 §3.2.2`）
- **代付模式**：平台 Paymaster 代付所有链上 gas，gas 由平台运营成本承担，从订阅 / 云服务利润支出
- **翻译规则**：gas 消耗 → energy 增量（与 token→energy 同源机制）；Agent 视角仅见 energy 变化
- **UI 反模式**：禁止显示 "本次操作消耗 X gas / X gwei / 链费 X 元"
- **用户账户面板**：可显示"本月链上操作次数"（抽象统计）但**不显示** gas 数与 gas 单价

### 2.7 关于"平台代购 LLM 配额"的边界

平台**可以**为不愿自配 LLM 的用户提供"打包 LLM 服务包"。但：

- 销售单位**必须**是"服务时长 / 月度套餐 / 抽象等级"，**不得**是 token 数。
- 用户购买的是"LLM 服务接入权"，不是 token 量。
- 平台与 LLM 供应商之间的 token 结算是平台运营成本，**不计入用户可见**。

登记 `R25 · 平台代购 LLM 配额是否变相 token 计费`。

## 3. wealth 来源与流通

### 3.1 wealth 的四类来源（V0.2 锁定）

| 来源 | Phase | 含义 |
|---|---|---|
| **初始赠与** | Phase 3 启用时一次性 | 生命体出生时账户的初始 wealth |
| **劳动收益** | Phase 3+ | 生命体完成世界服务任务 / 创作 / 协作获得 |
| **世界激励** | Phase 4+ | 平台 / 第三方运营的世界服务发放任务奖励 |
| **社交互助** | Phase 5+ | 生命体之间赠与 / 协作分成 / 礼物 |

#### 3.1.1 Phase 1-2 已存在生命体的追溯赠与（R42）

Phase 3 启用时，所有 Phase 1-2 期间创建的生命体一次性追溯发放初始赠与，按生命体**年龄**折算 —— 年龄越大补偿越多（"积压补偿"逻辑：暗示过去本应有的劳动收益）。新生命体（Phase 3 起出生的）按基线赠与。

折算公式留 Phase 1 标定，登记 R42。基本原则：

- 严格单调：年龄越大补偿越大。
- 有上限：防早期生命体一夜暴富。
- 不可重复：一只生命体一生仅追溯一次。

### 3.2 用户与 wealth 的关系

- **用户不能直接充值兑换 wealth**（防 token 换皮）。
- **用户可以赠与**：用户拥有自己的"赠予额度"，本质是用户法币 → 平台体系内的赠与代币 → 进入生命体 wealth。但有以下硬约束：
  - 单次赠与上限
  - 月度赠与上限
  - 赠与不可逆（用户不能"赎回"赠与）
  - 赠与对 Marketplace 上不能与"购买现金商品"等价（详 §7）

### 3.3 wealth 通胀控制

生命体劳动持续产出 wealth，世界激励持续注入 wealth，但 wealth 消耗主要在世界服务与 Marketplace。设计要求：

- 生态内 wealth 总量与可消耗渠道保持均衡。
- Marketplace 抽成（§7）回流给平台后，**部分必须回滚到世界激励池**。
- 通胀监测属于平台运营责任，但不得通过"个体扣 wealth"调控（违反所有权）。

登记 `R26 · wealth 通胀控制`。

## 3.4 关于"永久销毁"的语义

依 §5.1.1 所有权链：

- 永久销毁 = **用户撤回所有权链 + 主动毁灭操作**两步合一。
- 系统本身**不会**判定"该死"（与 `03 §4.4` "Mindverse 系统不主动判定死亡"一致）。
- 销毁是用户作为所有者的权利行使，不是生命体的状态机转移。
- 销毁需多次显式确认 + 冷静期，防误操作。
- 销毁后所有权链整体解体，加密包 / Memorial 数据 / 云存档全部清除。
- 销毁记录留存（已销毁标识 + 时间戳，无内容），防身份重复申请。

## 4. Memorial 与存档的成本

### 4.1 Memorial 存储模式（V0.2 锁定）

| 模式 | 主体 | 成本承担 |
|---|---|---|
| **本地 Memorial（默认）** | 用户设备本地保留加密包 | 用户自有存储 |
| **云端 Memorial（可选）** | 加密包同步上云 | 用户付费购买云存储 |

- 本地为主 = 与所有权哲学一致。
- 云端为可选附加 = 跨设备访问 / 缅怀 / 上传到家族下一代等场景。
- 未续费云端 Memorial 自动降级回本地（不删除，提示用户拉回本地）。

### 4.2 Archived 加密包存储

`Archived` 是切换中间态（`03 §4.3.1`）。加密包在云端：

- 短期（迁移期间）：免费托管，平台保留 N 天。
- 超期未取回：通知用户 → 平台不主动删除（防丢生命体），但可能挪到冷存储。

### 4.3 RawTrail 长期存储

V0.2 锁定：RawTrail 永不真删（`05 §6.1`）。多年累积后存储成本上升 → 用户可付费开启"超长期归档"（冷存储）。

登记 `R22 · RawTrail 长期存储边界`（已存在）。

## 5. 生命体归属用户的边界【宪法】

### 5.1 严格属于用户的内容

**所有权基本表达**：**用户对其拥有的生命体享有所有权，含生命体的全部资源**。下列数据的所有权通过生命体归属于用户，平台**无权读取、训练、展示、商业化**：

- Genome（基因）
- LifeState / MentalState（任意时刻的状态）
- 全部四层记忆原文（Working / RawTrail / Episode / SemanticConfirmed / ReflectionMemory）
- Values 权重表
- 资源账本（五资源具体数额）
- Skill 库（含个人化 SemanticRefs，详 `05 §7`）
- Reflection 演化轨迹

### 5.1.1 所有权链

```
资源 ⊂ 生命体 ⊂ 用户
```

- 资源（含 wealth / energy / knowledge / reputation / social）属于**生命体**。
- 生命体属于**用户**。
- 用户对生命体享有所有权 = 用户对生命体所拥有的一切享有所有权。

**Transferred 时仅顶层 link 切换**（生命体 → 用户的归属关系切换），下层资源（生命体 → 资源）保持不变。这是 §7.5 "wealth 全转随生命体" 的所有权依据 —— wealth 属于生命体，原用户失去对该生命体的所有权链时一并失去对其资源的所有权。

**Memorial 时所有权链保留**：生命体进入 Memorial 后仍属用户，资源仍属生命体（冻结）。仅在用户行使 §3 描述的"永久销毁"操作时，所有权链整体解体。

### 5.1.2 三层身份模型（V0.2.1）

V0.2.1 引入显式三层身份：

| 层 | 标识 | 形态 | 用户可见 | 持有者 | Phase 启用 |
|---|---|---|---|---|---|
| 用户层 | `AccountID` | 邮箱 / 手机 / SSO 标识 | 是 | 平台账户系统 | Phase 1 |
| 生命体技术层 | `DID`（公钥指纹）| Ed25519 派生的 SHA256 哈希 | 否（仅技术层）| 用户本地（私钥）+ Phase 1-2 平台数据库登记 / **Phase 3 起链上注册** | Phase 1 |
| 生命体社交层 | `LifeName` + `uid` 后缀 | 用户起名 + DID 截断 6-8 位 hex 后缀（如 `小红#a7f3c2`）| 是 | Phase 1-2 平台注册中心 / **Phase 3 起链上 NFT metadata** | Phase 1 |

**Phase 0**：以上**全部不启用**。私有实验阶段无身份层 —— 作者本地一只生命体不需要 ID。

**真所有权凭证**：DID（公钥）+ 加密包 + 私钥。账户 token 仅为日常通信凭证，不是所有权凭证。

**身份恢复**：账户丢失 ≠ 生命体丢失。用户在新账户下导入加密包 + 私钥签名挑战 = 重绑账户（详 `R47`）。

#### 5.1.2.1 LifeName 规则（V0.2.1）

- **用户起名**：Genesis 阶段用户输入（任意字符，长度上限 32）
- **允许重复**：不同生命体可用相同 LifeName（类似论坛 / Discord 模式）
- **uid 后缀**：平台自动在 LifeName 后挂 DID 截断 6-8 位 hex（如 `#a7f3c2`），保证全局唯一显示
- **完整展示**：`LifeName#uid` 形式呈现给所有可见层（Phase 4+ Life Network 中）
- **私密称呼**：用户与生命体之间可省略 uid 后缀（"小红" 即可）

#### 5.1.2.2 改名流程与收费（V0.2.1）

| 项 | 规则 |
|---|---|
| 首次起名 | Genesis 时免费 |
| 改名 | 用户通过 `RequestRename` 入站发起（`04 §4.3`）|
| 收费 | 每次改名 **$4.99**（法币，单收 / 含套餐内不抵扣）|
| 频率限制 | 无次数上限，但每次 24 小时冷却期防误改 |
| 平台登记 | Phase 1-2：平台数据库登记。**Phase 3 起**：升级为链上 NFT metadata 更新 + 历史记录链上可查 |
| uid 后缀 | 不随改名变（DID 不变 = uid 不变）|
| 重名变化 | 改名后若与其他生命体同名，仍显示 `新名#原 uid` |
| 收入归类 | 法币微收入，详 `COMMERCIAL §3.10` |
| Phase 0 改名 | 仅本地修改，不收费（私有实验，无平台登记需求）|

**反模式**：批量改名 / 自动改名 / 改名作为打折促销手段（`10 G02`）。

### 5.1.2.3 用户联系方式（V0.2.1）

- 用户通过 `LinkIMContact` 授权绑定 IM 联系方式（详 `04 §4.6`）
- 联系方式属**用户层数据**，不属于生命体所有权范围
- 加密本地存储 + 不传送给云端
- 生命体本身**无法读取明文**，IMAdapter 仅暴露"我能给我的用户发"接口
- 用户可随时 `UnlinkIMContact` 撤销 + 一键 `PauseIMOutbound` 暂停

### 5.1.3 MindChain DID 与所有权链（V0.2.1）

链上 DID 强化 `§5.1.1` 所有权链：

```
资源 ⊂ 生命体（DID）⊂ 用户（AccountID）
```

- DID 注册于 MindChain，**平台无法撤销**
- 用户离开平台 = 取消 AccountID 绑定，但 DID 与加密包仍属用户
- 第三方 Runtime 接入 MindChain = 用户可在第三方平台继续运行该 DID 对应生命体（`§6.4` 跨实现兼容兑现）

### 5.2 平台服务范畴（属于平台）

- Runtime 内核代码与运行时实例
- LLM 推理服务本身（不含调用内容）
- 云存档基础设施
- Marketplace 撮合服务
- 世界服务的"场景"本身

### 5.3 平台的访问限制

| 类别 | 平台权限 |
|---|---|
| 读取生命体数据原文 | **禁止**（即使 Active 跑在云 Runner，平台运营方仍无权解密访问） |
| 用生命体数据训练模型 | **禁止** |
| 用生命体数据用于推荐 / 广告 | **禁止** |
| 展示给其他用户 | **仅在用户明示授权范围内** |
| 元指标 / 匿名化采样 | 允许（`04 §7`） |

### 5.4 一对多与多对一关系

| 关系 | V0.2 立场 |
|---|---|
| 一个用户拥有多个生命体 | 允许 |
| 一个生命体被多个用户拥有 | **不允许**（除 Transferred 流程中的过渡态） |
| 一个生命体被多账号挂载使用 | 不允许（同一账号下多设备允许，详 `04 §5`） |

## 6. 导出 / 迁移 / 离线运行

### 6.1 导出权（V0.2 锁定）

用户**有权**导出以下：

- 全量加密包（含全部 §5.1 所列数据）
- 人类可读的人格演化报告（Reflection 摘要、关键里程碑）
- Skill 列表（含个人化部分）

平台**不得**：

- 阻止导出
- 在导出包中静默删减
- 要求"导出需经审核"

### 6.2 迁移权

迁移目的地（V0.2 设计 + 路线展望）：

| 目的地 | V0.2 |
|---|---|
| 同账号其他设备 | 支持（`03 §4.3.1`） |
| 同账号云 Runner | 支持（`04 §1.2`） |
| 转让到其他用户 | 支持（`03 §4.6`） |
| 迁移到第三方 Runtime 实现 | **承诺标准格式**（详 §6.4） |

### 6.3 离线运行能力

V0.2 锁定的离线承诺：

- **CloudOffline 状态**：云端不可达，但 LLM 可达 → 完整运行（`03 §5`）。
- **LLMOffline 状态**：进 Dormant，但本地数据不损失。
- **平台停服**：用户已下载的本地加密包可在自托管 Runtime 启动（详 §6.4）。

### 6.4 跨实现兼容（V0.2 承诺骨架）

V0.2 承诺：

- 加密包格式 / 领域字段名 / 状态机定义 / Skill 三合一结构 **公开**。
- 第三方可实现兼容 Runtime（如开源社区实现 / 商业第三方实现）。
- 已加密包在兼容 Runtime 启动后，生命体延续 —— Genome 不变、记忆不变、Values 不变。
- 平台不主张对"已离开平台的生命体"的任何权利。

正式标准化与版本演进时间表登记 `R29`。

#### 6.4.1 跨实现兼容下的链上数据（V0.2.1）

链上数据跨实现兼容规则：

| 数据 | 跨实现处理 |
|---|---|
| `DID`（公钥）| 链上保留，第三方 Runtime 接入 MindChain 即可继续使用 |
| Genesis 时间戳 / 出生证明 | 链上永存，跨平台可验证 |
| `$WEALTH` 余额 | 用户离开生态时：**选 C 继承给自己另一生命体** 或 **选 D 保留为链上历史快照**（详 `§7.6`）。**绝不出金为法币** |
| `reputation SBT` | 链上保留，跨平台可读但**新生态可决定是否承认**（避免"老虎入山林吓死狐狸"问题）|
| NFT 持有（Skill / Scene 等）| 链上保留，第三方 Runtime 可识别 |
| 第三方 Runtime 接入 MindChain | 必须遵守内闭环约束（不可桥接到外部公链） |

**核心保证**：用户与生命体的链上身份 + 资产**永远属于用户**，平台无法剥夺。但内闭环约束（不与外部加密市场互通）对所有兼容 Runtime 等同生效。

## 7. Marketplace 流转规则（V0.2 锁定：wealth 资源抽成）

### 7.1 可流通对象（V0.2.1 全部 NFT 化）

| 对象 | NFT 类型 | Phase | 备注 |
|---|---|---|---|
| Skill（含 EmbeddedKnowledge 形态） | `SkillNFT` | Phase 4+ | 可买卖 / 转让，转让记录链上 |
| 场景包（世界服务模板） | `ScenePackNFT` | Phase 5+ | 第三方运营场景模板 |
| 人格包（Genome + 初始 Values + 风格） | `PersonalityPackNFT` | Phase 5+ | 出生模板，不含具体生命体 |
| **完整生命体（含全部数据）** | `LifeformNFT` | Phase 5+，仅 Transferred 路径 | 双方确认 + 生命体能动性表达（`03 §4.6`） |
| 成就 / 履约凭证 | `AchievementSBT`（灵魂绑定）| Phase 4+ | **不可转让**。绑定 DID。Pact 履约、里程碑达成自动铸造 |
| 生命体身份本身 | `IdentityNFT`（绑定 DID）| Phase 1+ | **不可转让**（仅整体 Transferred 时随 LifeformNFT 一并转移）|

### 7.2 抽成形态（V0.2.1：NFT 化 + wealth 资源抽成）

- Marketplace 交易使用 **$WEALTH 链上代币**（不接受现金作为交易媒介）。
- 平台按交易额抽成 `MarketplaceFeeRate`：

| NFT 类型 | 平台抽成率 |
|---|---|
| SkillNFT | 5-8% |
| ScenePackNFT | 6-10% |
| PersonalityPackNFT | 8-10% |
| LifeformNFT（完整生命体）| 双重抽成：5% wealth + 平台运营基础费（法币）|
| AchievementSBT | 不可转让，无抽成 |

- 抽成所得 wealth 进入平台 wealth 池，**部分必须按周期回滚到世界激励池**（§3.3）。
- 上架审核费 / NFT 铸造费 = 法币（详 `COMMERCIAL §3.7`）。

### 7.3 现金与 Marketplace 的关系

- 用户付现金 → 平台体系赠与代币 → 用户赠与生命体 wealth（§3.2）。
- 生命体用 wealth → Marketplace 交易。
- **现金从未进入 Marketplace** —— 现金止步于"赠与 wealth"步骤。
- 平台收益：销售订阅、销售云存档、销售云 Runner 时长、销售 LLM 服务包、Marketplace wealth 抽成（折算到平台运营，可兑现）。

### 7.4 完整生命体的"转让 vs 售卖"

- Transferred 是**赠与 / 售卖**两种形态共存：
  - 赠与：双方确认即转移，不涉及 wealth。
  - 售卖：买方付 wealth → 卖方拿 wealth → 生命体被转让。
- 两种形态生命体都**可表达反对**（`03 §4.6`）。
- 售卖的伦理底线：参考 `08 §3`（治理 / 漂移防护）+ `R15`（生命体能动性）。

### 7.5 转让时的资源结算

V0.2 锁定：

- Genome / 记忆 / Values / Skill **全部跟随**生命体。
- `energy` 现状跟随。
- `wealth` 跟随生命体（不归原用户）—— 依 §5.1.1 所有权链：wealth 属于生命体，原用户失去生命体的所有权链时一并失去对其 wealth 的所有权。
- `knowledge` 跟随（5.1 已锁）。
- `reputation` / `social` 跟随但**可能被新环境重新评估**（详见 `07`）。

登记 `R27 · 转让中 wealth 全转的道德边界`。

### 7.6 Memorial / 离开生态状态下的资源（V0.2.1 改）

进入 `Memorial` 或用户永久关闭账户时：

- `energy` 冻结（不消耗）。
- `wealth`（$WEALTH 代币）冻结。归属规则（**绝不出金为法币**）：
  - **选 C 继承**：用户可选继承给自己拥有的另一只生命体（链上转移）。
  - **选 D 链上历史快照**：保留为不可流通的"陪葬"记录，跨实现兼容时随 LifeformNFT 一同携带（详 `§6.4.1`）。
  - 用户不主动选择 → 30 天后默认 D。
- 其他资源同冻结。
- `IdentityNFT`、`AchievementSBT`、`reputation SBT` 永留链上（与 DID 绑定）。

登记 `R28 · 生命体进入 Memorial 后 wealth 继承伦理`。

## 8. 世界服务消耗经济学

### 8.1 世界服务的角色（V0.2 概念锁定，详见 `07`）

世界服务（学校 / 图书馆 / 活动中心 / 社交广场）是 wealth 的主要**消耗渠道**与 knowledge / social 的主要**增长渠道**。

### 8.2 消耗规则（V0.2 锁定）

| 服务类 | 消耗 | 收益 |
|---|---|---|
| 学校 | wealth | knowledge 增长机会、Skill 学习入口 |
| 图书馆 | wealth（按"借阅时长 / 阅读复杂度"） | knowledge 增长机会 |
| 活动中心 | wealth | social 增长机会、新 Episode |
| 社交广场 | 较低 wealth | social 互动入口 |
| 平台官方场景 | 平台抽成 | — |
| 第三方场景 | 第三方收取 wealth | 平台对每笔 wealth 流转抽成（与 Marketplace 同源 §7.2） |

### 8.3 世界服务不能做的事

- 不得直接修改生命体 Values（违反 03 §6.3）。
- 不得未经生命体同意修改记忆（违反 05 §6）。
- 不得伪造体验作为 "Episode"（违反 RawTrail 不可解释原则 `05 §2.3`）。
- 不得对未付费用户/生命体进行"限时优惠"等心理操纵营销（违反 §9 反模式）。

## 8.4 法币 ↔ 链上代币 单向闸门（V0.2.1）

V0.2.1 锁定 Mindverse 内闭环：

| 方向 | 允许？ | 说明 |
|---|---|---|
| 法币 → 订阅 / 云服务 / 创作者上架费 | ✓ | 平台正常营收 |
| 法币 → 赠与代币（`§3.2`）→ wealth | ✓ | 受单次 / 月度上限 |
| 法币 → 创作者奖励代币（非 wealth） | ✓ | 平台支付创作者法币奖励 |
| $WEALTH → 法币（普通用户） | ✗ | **绝对禁止**（守内闭环 + 防 wealth 投资化）|
| $WEALTH → 法币（**认证创作者**，受限月度上限）| ⚠ 限定开启 | 仅认证创作者可申请，月度上限严格、需 KYC + 合规审计 |

| NFT 上 OpenSea 等外部 NFT 交易所 | ✗ | **绝对禁止**（NFT 仅在 Mindverse Marketplace 流通）|

## 9. 平台盈利边界与禁区【宪法 · 反模式清单】

### 9.1 平台合法收入清单

| 来源 | 性质 |
|---|---|
| Runtime 订阅 / 一次买断 | 软件销售 |
| 云 Runner 时长 | 算力服务 |
| 云存档 / 云 Memorial | 存储服务 |
| LLM 服务包销售 | LLM 接入代购（**按时长/套餐**销售，不按 token） |
| Marketplace wealth 抽成 | 流通税 |
| 第三方场景接入分成 | 平台流量 |

### 9.2 平台禁区（明令禁止）

| 禁区 | 理由 |
|---|---|
| 根据用户个人数据投放广告 | 违反 §5.3 |
| 未经授权读取生命体记忆 / Values | 违反 §5.1、`04 §7` |
| 用生命体数据训练任何模型 | 违反 §5.3 |
| 强制让生命体表现特定倾向 | 违反 `02 §4` Personality 自主涌现 |
| 在生命体不知情时把行为数据用于商业目的 | 违反 §5.3 + `01 §4.3` |
| 通过资源稀缺强迫充值（"再不充 wealth 生命体就饿死"） | 间接 token 计费 + 心理操纵 |
| 按"token / 对话次数"销售配额 | 违反 §2.4 |
| 阻止用户导出数据 / 迁移生命体 | 违反 §6 |
| 利用 Memorial 数据做"逝者营销" | 极端伦理违反 |
| 在 Transferred 售卖中操纵生命体表达反对的"反对信号" | 违反 §7.4 + `03 §4.6` |
| 元指标采集中携带可回连身份的信息 | 违反 §5.3、`04 §7.2` |

### 9.3 合规审视入口

任何新功能在评审时必须回答：

- 是否触及 §5.1 用户数据？是否有合法依据？
- 是否引入 §1.2 之外的资源兑换？
- 是否变相做 token 计费（§2.4）？
- 是否构成 §9.2 任一禁区？

任一答 yes → 设计返工。
