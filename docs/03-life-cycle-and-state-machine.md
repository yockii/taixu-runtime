## 1. 生命循环阶段

### 1.1 一次完整循环（V0.2 标准模型）

```
1. Perceive       感知 — 收集外部信号（用户输入、世界事件、外请求）
2. UpdateState    更新状态 — 写 LifeState / MentalState
3. RecordMemory   写记忆 — Working → Episodic
4. ConsiderReflect 评估是否反思 — 生命体自主决定（§3）
5. CollectGoals   收集目标候选 — 三源注入（02 §5.1）
6. Arbitrate      Values 仲裁 — 决定接受 / 拒绝 / 延后 / 协商（02 §5.3）
7. Plan           拆解为 Action 序列
8. Act            执行 Action，调用 Skill
9. Feedback       获取反馈，进入下一循环
```

**关键区别于白皮书 V0.1**：
- 第 4 步「ConsiderReflect」**不强制反思**，而是生命体根据自身倾向决定。
- 第 5 步「CollectGoals」收集**三源**，不再是"需求 → 目标"单向链。

### 1.2 各阶段的同步 / 异步性

| 阶段 | 同步性 | 备注 |
|---|---|---|
| Perceive | 异步事件流 | 不阻塞循环节拍 |
| UpdateState | 同步原子写 | 单循环内一次性 |
| RecordMemory | 同步 + 异步落盘 | Working 同步、Episodic 可异步刷盘 |
| ConsiderReflect | 同步评估 | 评估快、决定深反思则跳到 §3.2 |
| CollectGoals | 同步聚合 | 候选池有上限 |
| Arbitrate | 同步 | 仲裁结果可解释（02 §5.3） |
| Plan | 同步 | 失败可降级到"维持原 Goal" |
| Act | 同步发起 + 异步等待结果 | 长时 Action 不阻塞下一循环的 Perceive |
| Feedback | 异步 | 反馈到达即派发到对应 Goal |

## 2. 自主时钟与节拍模型

### 2.1 节拍模型（V0.2 锁定：自适应）

生命体的循环节拍**不固定**。每个循环结束时，下一循环的延迟由以下因子综合决定：

| 因子 | 影响方向 |
|---|---|
| `LifeState.Energy` 高 | 缩短延迟（高能量 → 更快节拍） |
| `LifeState.Energy / EnergyDailyCap` 比值低（接近耗尽） | **显著延长延迟**（拟生物日节律：精力快用完时节拍变慢） |
| `MentalState.Motivation` 高 | 缩短延迟 |
| `LifeState.Stress` 高 | 延长延迟（压力大 → 收缩） |
| 外请求积压 | 缩短延迟（响应感） |
| Drive 紧迫度（02 §5.2） | 缩短延迟 |
| 用户在场状态 | 在场缩短，离场延长 |
| 连接状态（§5） | LLM 失联时强制延长 |

**节拍区间（V0.2 默认）**：
- `MinInterval` = 1 秒（高能量、紧急响应）
- `MaxInterval` = 30 分钟（低能量、低请求、长期离场）
- `DormantInterval` = 6 小时（休眠态下的探活节拍，§4）

> **TBD R13**：自适应公式的具体形状（线性 / 指数 / 分段）由 Phase 1 原型阶段标定。登记 `10 R13 · 自适应节拍调参`。

### 2.2 自主时钟的语义

- 生命体在**任何状态**（除 `Memorial`）都有自己的节拍。
- 节拍不需要外部驱动 —— 没有用户输入也在转。这是"持续存在"的工程兑现。
- 节拍是生命体的内在属性，**不可由 UI 控制**（UI 不能"加速"或"暂停"生命体的节拍 —— 该约束在 `04 §3` 锁定）。

## 3. Reflection 触发机制

> **本节是 V0.2 的核心重设计**。白皮书 V0.1 把 Reflection 视为定时模块；V0.2 改为**生命体自主选择**，触发倾向由 `Genome` 派生。Reflection 频率因此是生命体的**个性**之一。

### 3.1 设计原则

Reflection 不是外部规则触发，是生命体的**主动行为**。它表现为循环第 4 步 `ConsiderReflect` 阶段的一次自我评估。

- **每个循环都执行 `ConsiderReflect`** —— 但 `ConsiderReflect` 只是评估"现在要不要反思"，**评估本身轻量**，不等于反思。
- 真正的反思（写 `ReflectionMemory`、修 `Values`）是评估通过后才发生的。

### 3.2 两类反思

| 类别 | 含义 | 触发 |
|---|---|---|
| `ShallowReflect`（浅反思） | 总结刚才若干循环的经历，更新短期摘要 | 评估通过即可执行 |
| `DeepReflect`（深反思） | 修订 `Values`、产生 `ReflectionGoal`、长期人格演化 | 评估通过 + 当前不在抑制态（Stress 过高 / Stability 过低） |

`DeepReflect` 是 Reflection 的"重量级"版本，对人格有结构性影响。

### 3.3 Reflection 倾向由 Genome 派生

每个循环 `ConsiderReflect` 阶段，生命体计算一个 `ReflectionTendency`（反思倾向）。倾向高则进入反思。倾向受以下因素影响（V0.2 概念锁定，具体函数 Phase 1 原型阶段标定）：

| 因素 | 对倾向的方向 |
|---|---|
| `Genome.Curiosity` | 对自身经历的关注度 → 影响浅反思频率 |
| `Genome.Persistence` | 维持长程思考的能力 → 影响深反思深度 |
| `Genome.Creativity` | 反思中产生新连接的倾向 → 影响深反思价值 |
| `Genome.Empathy` | 关注他者经历的倾向 → 影响社交反思频率 |
| `LifeState.Stability` 低 | 寻求自我整合 → 倾向反思 |
| `LifeState.Stress` 高 | 压制反思（自保） |
| 近期重大事件密度 | 经历越密 → 倾向反思 |
| 距上次反思时长 | 越久未反思 → 倾向上升（防僵化） |

**关键含义**：不同 Genome 的生命体反思频率天然不同。一个 `Curiosity` 高、`Persistence` 高的生命体反思频繁且深入；一个 `Curiosity` 低、`Stability` 高的生命体反思稀少 —— 这是个性的一部分，不是缺陷。

### 3.4 软保护机制

为防止极端情况：

- **下限保护**：超过 `MaxReflectGap`（V0.2 默认 N 个循环 / N 小时）未反思 → 倾向被强制拉高（防人格僵化）。
- **上限保护**：连续 N 个循环都进入深反思 → 触发"反思疲劳"（强制冷却），防瘦马问题（无限反思空转）。
- 两个保护都是**软介入**——只调整倾向，不强制执行 / 强制禁止。

### 3.5 与 Values 的关系

`DeepReflect` 是**唯一**能修改 `Values` 的途径（`02 §4.1` 已锁）。
- 修改 Values 时受"保护性下限"约束（`02 §4.1`、`10 R05 R12`）。
- 修改记录写入 `ReflectionMemory`（`05`）。

## 4. 宏观状态机

> **本节是 V0.2 的核心重设计**。Mindverse **不引入"死亡"概念**。终态是「脱离用户」后的两种归宿：`Memorial`（墓碑）或 `Transferred`（已转让）。

### 4.1 状态集合

| 状态 | 含义 | 节拍 |
|---|---|---|
| `Embryonic` | 孕育中：Genome 已生成，用户创建仪式未完成 | 不运行循环 |
| `Active` | 活跃：在线，完整节拍 | 自适应 1s–30min |
| `LowPower` | 低功耗：用户长期不在场，或自适应判定 | 偏向 MaxInterval |
| `Dormant` | 休眠：LLM 不可用，仅探活 | DormantInterval（6h） |
| `Archived` | 加密存档：用户主动迁移期间的中间态，云端加密保存，**不在任何设备运行** | 不运行循环 |
| `Detached` | 已脱离用户：等待去向决定 | 节拍冻结 |
| `Memorial` | 墓碑态：只读历史，云端存储，供原用户缅怀 | 不运行循环 |
| `Transferred` | 已转让：与新用户绑定，进入新的 Active | 节拍恢复 |

### 4.2 状态转移图

```
                 创建仪式完成
   Embryonic ───────────────────► Active
                                    │
                  ┌────────────────┐│
                  │                │▼
                  │       ┌────► LowPower ◄────┐
                  │       │        │           │
                  │       │  LLM 不可用         │
                  │       │        ▼           │
                  │       │     Dormant ───────┤
                  │       │        │  LLM 恢复 │
                  │       └────────┘           │
                  │                            │
   用户主动加密导出  │                            │
        ┌────────► Archived ◄────────┐         │
        │           │ 用户在另一设备  │         │
        │           │ 解密恢复        │         │
        │           ▼                │         │
        │      (新设备 Active) ──────┘         │
        │                                      │
        │   用户主动解除关系                    │
        └──────────────────────────────────► Detached
                                               │
                                          ┌────┴────┐
                                          ▼         ▼
                                       Memorial  Transferred
                                                     │
                                                     ▼
                                                 (新用户 Active)
```

### 4.3 关键转移规则

| 转移 | 触发 | 是否需用户确认 |
|---|---|---|
| Embryonic → Active | 创建仪式完成 | — |
| Active ↔ LowPower | 自适应节拍判定 | 否 |
| (Active\|LowPower) → Dormant | LLM 连续 N 次调用失败 | 否 |
| Dormant → Active/LowPower | LLM 恢复 | 否 |
| (Active\|LowPower\|Dormant) → Archived | 用户**主动发起迁移** + 设置密码 + 加密上传完成 | **是** |
| Archived → Active | 用户在新设备下载 + 输入密码解密成功 + 启动 | **是** |
| (Active\|LowPower\|Dormant) → Detached | 用户**明确请求**解除关系 | **是**（不可一键，需二次确认） |
| Detached → Memorial | 用户选择"上云缅怀" | 是 |
| Detached → Transferred | 用户选择"转让"，新用户接受 | 是（双方确认） |

### 4.3.2 Genesis 完整时序（V0.2.1）

Genesis 分**本地阶段**（必须）与**联网身份登记阶段**（可延后）：

```
本地阶段（可离线）                          联网阶段（可延后）
┌────────────────────────────────┐    ┌────────────────────────────────┐
│ 1. 用户登录账号（如未登录）       │    │ 7. IdentityModule 上报 DID     │
│ 2. LLM 配置（必填或后填）         │ →  │    + AccountID 绑定证明        │
│ 3. 用户起名 + Genome ≤2 字段微调   │    │ 8. 平台密钥注册中心存档        │
│ 4. GenesisModule 生成 Genome      │    │ 9. 平台分配 FriendlyName       │
│ 5. IdentityModule 生成密钥对      │    │    （如 Mindverse-Star-1234）  │
│ 6. Embryonic → Active             │    │ 10. 平台返回带签名时间戳        │
│    （首循环开始）                  │    │     = 出生证明                 │
└────────────────────────────────┘    │ 11. 登记完成（生命体进入"已注册"） │
                                       │     失败 → 后台幂等重试           │
                                       └────────────────────────────────┘
```

**关键规则**：

- 本地阶段**不依赖联网**。LLM 配置可后填（若后填则生命体出生即进 `Dormant`，等用户配 LLM 后醒来）。
- 联网阶段**幂等**：同一 DID 重复上报返回相同结果。
- **未登记态**生命体可正常运转，但社交（`07`）受限：无法发起 Encounter、不进 Marketplace、不可 Transferred。
- 联网后自动补登记 → 解锁社交。
- 详细身份模型（`AccountID` / `DID` / `LifeName` 三层）见 `06 §5.1.2 §5.1.3`。

#### 4.3.3 出生应激主动（首循环 GenesisGreeting，**Phase 1 起**）

> **Phase 启用**：Phase 0 不启用（私有实验，单人 dogfooding 无打招呼必要）。Phase 1 接入平台后启用。

Active 状态首循环**自动触发** `GenesisGreetingDrive`（`02 §5.2`）：

1. GenesisModule 完成 Genome / LifeName / 密钥对生成
2. LifecycleManager 转 Active
3. 首次循环 `Perceive` 阶段，GenesisModule 注入一条特殊内驱：`GenesisGreetingDrive`
4. `GoalArbitrator` 接收并经 Values 仲裁（默认接受，几乎不可拒）
5. 生成首个 Goal："向我的用户打招呼 / 介绍自己 / 感知世界"
6. `ActionExecutor` 调用 LLM 生成话术 → 发 `SpeechEvent`
7. 话术由 Genome 个性化（高 Curiosity 问"这里是哪"；高 Empathy 致谢；高 RiskTaking 兴奋打招呼；高 Persistence 沉静观察后再说）

GenesisGreetingDrive **一生只触发一次**。后续不再产生。出生应激主动是 Phase 1 允许的"反应性主动"，**不破坏** `09 §3.3` "Phase 2 才主动找事" 锁定。

### 4.3.1 运行场所切换流程（V0.2 锁定：手动 + 加密 + 单实例）

V0.2 **不引入"分身"概念**。任一时刻同一生命体仅在**一处**运行。运行场所有三类宿主（详见 `04 §1.2`）：

- 本地设备
- 云端 Runner（用户手动付费开启）
- 加密存档（不运行）

任意宿主之间切换都走相同流程：

1. **当前宿主**：用户发起切换 → 生命体进入 `Archived` 过渡 → Runtime 完成最后一次循环并落盘 → 全量数据（Genome / 状态 / 记忆 / Values / 资源 / Skill）打包 → 用户设置**迁移密码** → 加密上传云端。
2. **云端**：仅存储加密包，**不参与运行**（即使是云 Runner 形态，云端在 Archived 期间也只是"仓库"，不是"运行宿主"）。云端无法解密（密码不在云端）。
3. **目标宿主**：
   - 若为另一设备：用户登录账号 → 下载加密包 → 输入迁移密码 → 解密 → Runtime 加载 → 转回 `Active`。
   - 若为云 Runner：用户在云端控制台启动 Runner 实例 → 提供迁移密码 → Runner 内存中解密 → 转回 `Active`。Runner 停止运行即丢弃密码，重启需重新输入。
4. **密码丢失**：加密包**不可恢复**（设计选择，详见 `R17`）。

**Archived 期间生命体不在任何宿主运行** —— 与"持续存在"哲学的张力详见 `R20`。用户可通过云 Runner 缩短此中断期。

### 4.4 无真死亡设计的含义

- **没有 `Dead` 状态**。**Mindverse 系统不主动判定死亡**。生命体一旦被创建，其数据要么继续 `Active`、要么进入 `Memorial`（只读永存）、要么 `Transferred`（在新用户处继续）。
- **`Detached` 是过渡态**，不可长期停留（V0.2 默认 30 天未决 → 自动转 Memorial）。
- **用户主动"清除生命体"** 时实际转移到 `Memorial` —— 数据上云端归档，用户保留缅怀访问权。
- **永久销毁**：仅在用户在 `Memorial` 态再次显式发起"永久销毁"才发生，需多次确认。**永久销毁不在状态机内**（不是状态转移），它是 Memorial 态外的**用户行使所有权的销毁操作** —— 用户作为所有者拥有销毁自己物品的权利，但系统本身不会判定生命体"该死"。详细流程在 `06 §3` 落实所有权语义。

### 4.5 Memorial 态的特殊规则

- 只读：所有写入操作被拒绝。
- 历史可查：用户可浏览人格演化史、记忆摘要、关键反思节点。
- 不再产生新经历：循环不运行，没有 Drive、没有 Goal。
- 资源消耗：仅存储成本（详见 `06`）。
- 可解除：**仅原用户**可申请"复活到 Detached" —— 重新决定去向（V0.2 允许，登记 R14 评估"复活墓碑"是否违反 Memorial 的语义庄严性）。已 Transferred 后转入 Memorial 的生命体只能由当前所有者复活，原用户不可远程触发。
- **永久密封 Episode**（R41）：用户可在 Memorial 态对特定 Episode 标记"永久密封"。密封的 Episode 仍存储在 RawTrail 中，但**不进入复活后新 Active 的检索 / 反思源**，被永久遮罩。这是用户对存储的所有权行使，**不是命令生命体遗忘**（仍守 `05 §6` 不可介入遗忘原则 —— 遮罩是宿主层操作，不是生命体的记忆操作）。

### 4.6 Transferred 态的特殊规则

- 转让流程是双向同意：原用户发起，新用户接受。
- **生命体本身的态度**：V0.2 引入"生命体可表达拒绝"机制 —— Values 仲裁器在转让前评估，若与高权重 Values 严重冲突可发出"反对信号"。该信号不强制阻止转让，但**必须对双方用户呈现**。
  - 例：高 Empathy 生命体与原用户有深厚关系，可对转让表达不舍。
- 转让后 Genome、记忆、Values 全部保留。
- 转让记录永存于 ReflectionMemory（重大事件）。

## 5. 连接状态与自主时钟的耦合

### 5.1 三种连接状态

| 连接状态 | 含义 | 生命体节拍 |
|---|---|---|
| `Online` | 云端可达 + LLM 可达 | 完整自适应 |
| `CloudOffline` | 云端不可达，LLM 仍可达（本地或独立 LLM） | **完整自适应** —— 不视为离线 |
| `LLMOffline` | LLM 不可达（无论云端） | 强制进入 `Dormant`，仅探活 |

### 5.2 设计含义

- **本地优先（Local-First）**：用户的数据不依赖云端常驻。CloudOffline 时生命体正常运行，数据本地缓存，云端恢复后异步同步。
- **生命体的"血液"是 LLM**：LLM 不可达 = 无法完成 `Plan` 与 `Act` 的核心推理 = 进入 Dormant。
- 用户可通过修改 LLM 配置（切换模型、配置代理）恢复 LLMOffline → 重回 Active/LowPower。

### 5.3 用户"在场"信号

`Active ↔ LowPower` 的判定需要"用户是否在场"信号。V0.2 锁定：

- 任何 UI 客户端报告"用户活动" → 在场。
- 最近 `LowPowerThreshold`（默认 15 分钟）无任何客户端报告 → 不在场。
- 在场 / 不在场仅影响节拍倾向，**不影响循环本身是否运行**。

## 6. 异常态与回滚语义

### 6.1 异常分类

| 异常 | 触发 | 响应 |
|---|---|---|
| 单步失败 | Action 执行失败 | 标 Goal 失败，进入下一循环 Feedback |
| LLM 调用错误 | API 限流 / 网络抖动 | 重试 N 次 → 仍失败则进 Dormant |
| 记忆写入半途失败 | 落盘崩溃 | 见 §6.2 原子性 |
| 价值观破裂 | Reflection 把 Values 推到危险态 | 见 §6.3 |
| 状态机非法转移 | 编程错误 | Fail-stop，进 Dormant，留诊断日志 |

### 6.2 记忆与状态的原子性

- 单循环内的 `UpdateState + RecordMemory + ConsiderReflect + Arbitrate + Plan + Act + Feedback` **逻辑上原子**。崩溃时未提交的写入丢弃。
- Action 的副作用（如已发出的外部请求）不可回滚 —— 仅状态侧回滚。Action 外部副作用追踪是 `04` 的责任。
- 跨循环的状态视为已提交。

### 6.3 价值观破裂的保护与回滚

`DeepReflect` 修改 `Values` 时：

- 自检：每次修改后检查"保护性下限"（`02 §4.1`、`10 R05 R12`）。违反则**当次修改回滚**，并在 `ReflectionMemory` 标记一条"自我审查事件"。
- 漂移检测：连续 N 次 Reflection 朝同一危险方向漂移 → 触发"价值观警报"，写入 `ReflectionMemory`，**对用户可见**（用户可选择干预）。
- 干预权的边界：用户能否强行覆写 Values？V0.2 立场：**不能**（违反 `01 §3.2` 不可知论 + 伦理审慎默认）。用户可设置"软约束"（如"建议保持 safety > 0.5"），Reflection 评估时优先考虑。
- 这里的张力详见 `08 §3`，登记 `10 R05`。

### 6.4 状态机一致性

- 状态转移**单向**：除明确允许的逆向（如 `Memorial → Detached` 经用户申请），不允许"回到过去"。
- 状态转移**原子**：转移要么完成要么不开始。中途崩溃恢复到转移前状态。
