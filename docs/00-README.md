## 三、文档体系的纪律

### 3.1 基石"一处定义，多处引用"

只有两份文档允许**定义**核心概念，其他文档只能**引用**：

- **基石①** `02-glossary-and-domain-model.md` — 所有术语、领域模型字段、不变量
- **基石②** `06-resource-economics-and-ownership.md` — 五资源语义、token 不暴露原则、生命体所有权边界、平台盈利禁区

发现某概念在非基石文档被"重新定义"或"扩展含义"时，是设计纪律事故，应把变更迁回基石文档，其他文档改为引用。

                └────────────┬──────────────┘

## 四、替代组织方案及其排除理由

设计本目录前考虑过另外两种组织方式，均被排除。记录理由防止未来重提：

### 方案 B：按四层架构切（UI / SDK / Runtime / Storage 各一份 + 横切若干）

**优点**：契合四层架构图，工程师友好。

**排除理由**：
- 领域模型（`02`）、宪法（`06`）这类横切内容会被切碎到四份文档中重复出现，违反"一处定义"纪律。
- 项目处于概念阶段，分层细节尚不稳定，按层切容易过早固化结构。

### 方案 C：按六阶段切（Phase 1 一份 ... Phase 6 一份 + 总览）

**优点**：路线图叙事强，对投资人 / 合作方友好。

**排除理由**：
- 同一模块（记忆、Reflection、资源系统）会在多个 Phase 文档中反复出现，难以维护单一真相。
- 用户本轮锁定的四维（领域模型 / 生命循环 / 模块边界 / 数据所有权）会被阶段叙事打散，每维难以聚焦。

### 当前方案（按关注点拆分多文档）的取舍

- **代价**：文档数偏多（11 份），需要本 README 作索引引导；新读者需要先理解依赖图才能选择起点。
- **收益**：四维有专属基石文档；六阶段由 `09` 单点横切回链；新增章节不撑爆单文档；演化时基石稳定、其他文档可独立迭代。

## 五、与白皮书 V0.1 的关系

- 白皮书 V0.1 是**愿景输入**，本目录是**设计骨架**。
- 当本目录与白皮书 V0.1 出现冲突时，**以本目录为准**；同时把冲突登记到 `10-risks-and-open-questions.md` 的"V0.1 → V0.2 演进留白"。
- 白皮书未来会迭代到 V0.2 — 届时本目录的成熟章节将反向回灌到白皮书。

## 六、给未来 Claude 实例的提示

1. 起草任一份文档**正文**前，先读 `02` 与 `06`（基石）；若该文档章节会涉及术语或资源/归属，必须回链而不重新定义。
2. 用户讨论新概念时，先判断它属于基石还是衍生 — 若属于基石，应该写入 `02` 或 `06` 并通知其他文档更新引用，**不要**就近写入当前讨论的文档。
3. 当用户说"新增 Phase X 的某能力"，先查 `09` 的阶段矩阵，再查相关基石是否已定义。
4. 任何"运行 / 测试 / 启动开发服务器"指令在本仓库目前都不适用 — 仓库无代码。
